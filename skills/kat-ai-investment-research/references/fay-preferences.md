## 判断维度与权重

### 八维度评估框架（投资评论专用，见 SKILL.md）
产品 Sharpness（最高）/ Bottom-up vs Top-down / 创业 Pattern 匹配 / 战略 Disruptiveness / 壁垒真实性 / 胜率×赔率 / 竞争格局 / Meaningful Stake

### 通用思维模式（跨场景，评论/访谈/研究都适用）

1. **感受 vs 产出，强制区分**：不接受笼统的“有帮助”“用起来不错”，必须拆成两问——「主观体验上是否更顺、更有掌控感」和「客观产出/质量/速度是否真的提升」。二者混为一谈是典型减分项。
2. **拒绝空谈理论，只认具体 case**：任何抽象判断（tradeoff、可替代性、价值高低）都会追问“能举个你自己的例子吗”“具体是哪件事”。给不出具体案例的判断，权重直接归零。
3. **精准的概念区分**：习惯把看似同义的概念拆开定义并验证是否命中，例如“Agent 协作平台”（有真实分工、结果互相传递）≠“多 Agent 平台”（多个 Agent 各自独立、平行运作）。样本/用户选择要卡在精确定义上，而非表面相似。
4. **产品直觉式归因，反推到机制层面**：现象级问题（如“用户不信任 Agent”）不停留在能力/体验层面的表面解释，而是反推到更底层的产品机制原因（如“不显示 runtime → 默认认为在偷懒”，即可见性问题而非能力问题）。
5. **主动质疑可替代性**：对一个产品/功能/定位，会习惯性追问“这个东西是不是本质上可以被别的方案替代”（例如指出某个多 Agent 产品“更多是一个 Personal Agent，可能确实可以被替代”）。
6. **样本代表性要求可量化**：判断一类用户是否“典型”，不满足于定性描述，会追问可验证的量化信号——长期依赖的粘性、实际分配的时间、具体使用 case、消耗的 token 量等。

## 加分清单

- 能清楚区分“主观感受”与“客观产出”两个维度，并各自举出具体例子
- 能用一个完整、可追溯的具体 case 来支撑判断（而非泛泛而谈）
- 对核心概念（如“协作”vs“并行”）有精确认知，且用来筛选/定义样本
- 能提供可量化的粘性/使用数据（时长、频率、token 消耗、留存）
- 主动思考并回答“这个东西是否可被替代”而不回避

## 减分清单

- 把“主观体验好”直接等同于“产品有价值”，不区分感受和产出
- 用理论/感受代替具体案例，问三次也讲不出一个实际发生的事
- 把“多个 Agent 平行运作、各自独立回复”误认为是“Agent 协作”
- 遇到用户不信任/不采用的现象，只归因到“功能不够强”，不深挖产品机制层面的原因（如可见性、透明度）
- 样本选择停留在表面相似（比如都叫“开发者”），不看长期依赖度、真实分配时间等更硬的指标

## 历史案例

### Slock/Raft 用户访谈提纲 Review（2026-07-07）

背景：团队准备了一份 40 分钟版用户访谈提纲，用于访谈 Slock（多 Agent 协作产品，后改名 Raft）的极客/普通/流失用户。Fay 给出改进反馈，核心指向两类问题：**用户方向不够典型** + **访谈追问深度不够**。

反馈要点（详见月报原话摘录）：
- 用户侧：应优先找“有协作需求的小团队”（核心 power user）和更多“独立开发者”，且现有开发者样本的粘性/时间分配/case/token 数据都不够典型。
- 追问侧：五个具体改进点——(1) 感受 vs 产出要分开追问；(2) 单/多 Agent 的 tradeoff 必须举例，不能只谈理论感受；(3) Compacting 场景下单 Agent 续跑 vs 拆分多 Agent 的判断逻辑，要具体案例；(4) 精准指出用户用的其实是“多 Agent 平台”而非“Agent 协作平台”，选人要卡住这个区分；(5) 把“不信任 Agent”归因到“看不到 runtime 显示 → 默认认为在偷懒”这一更底层的产品机制，而非停留在能力层面。

应用：这套反馈体现的思维模式已抽象进“通用思维模式”一节，可用于预判 Fay 在类似用户研究/产品定性材料评审场景下的关注点。

## 月报原话摘录

### 2026-07-07 · Slock/Raft 用户访谈提纲反馈

> 关于用户，我觉得核心还是今天我们刚刚讨论到的那几个点，多找一些这个方向的用户：
> 1. 多找一些有协作需求的小团队：这可能是我们最核心的 power user。
> 2. 开发者：我们目前访谈下来的开发者还是比较少的，而且很多是独立的开发者，独立在用 Slock 做一些简单的 research。我觉得长期依赖的粘性、实际分配的时间、在 Slack 上用的 case，以及花的 token 都不足够典型。

> (1) 在访谈侧，我们要追问得更细节一些，感受上好，还是产出上更有帮助；(2) 在这种 actually 的 tradeoff 是单 agent 还是多 agent 来做，最好这个时候要举例子，而不只是谈理论感受 (3) Compacting 怎么做？到底应该选择单 Agent 还是多 Agent 方案？目前看 Hermes 更多是一个 Personal Agent，可能确实可以被替代。(4) 我觉得本质上，这些用户用的不是“Agent 协作平台”，而是“多 Agent 平台”，我们要找一些 Agent 协作的人。这个观点我觉得说得很有道理

> (5) 理解信任问题：有时候觉得是不信任 agent 的原因是因为不会主动推送信息，所以 agent 会偷懒；只要他们不显示 runtime 就默认会偷懒
